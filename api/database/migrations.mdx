FixTweetBot uses [Masonite ORM](https://orm.masoniteproject.com/) for database migrations. This guide covers how to run migrations and understand the migration system.

## Running Migrations

To initialize or update the database schema, run the following command from the project root:

```bash
masonite-orm migrate -C database/config.py -d database/migrations
```

<ParamField path="-C" type="string">
  Path to the database configuration file (`database/config.py`)
</ParamField>

<ParamField path="-d" type="string">
  Path to the migrations directory (`database/migrations`)
</ParamField>

<Note>
  Make sure your database connection is properly configured in `override.config.yml` before running migrations.
</Note>

## Database Configuration

The database configuration is loaded from `database/config.py`, which reads settings from your `override.config.yml` file:

```yaml
database:
  host: <your_database_host>
  driver: <your_database_driver>  # mysql, postgresql, sqlite, etc.
  database: <your_database_name>
  user: <your_database_user>
  password: <your_database_password>
  port: <your_database_port>
```

**Supported drivers:**
- `mysql` - MySQL/MariaDB (recommended)
- `postgresql` - PostgreSQL
- `sqlite` - SQLite
- `mssql` - Microsoft SQL Server

## Migration File Naming

Migration files follow a timestamp-based naming convention:

```
YYYY_MM_DD_HHMMSS_description.py
```

**Example:**
```
2023_11_17_092808_guilds_table.py
2024_06_24_175057_members_table.py
2025_03_16_211708_create_events.py
```

This ensures migrations run in chronological order.

## Migration Best Practices

<Steps>
  <Step title="Backup First">
    Always backup your database before running migrations, especially on production.
    
    ```bash
    # Example for MySQL
    mysqldump -u user -p database_name > backup.sql
    ```
  </Step>
  
  <Step title="Test in Development">
    Test migrations in a development environment before applying to production.
  </Step>
  
  <Step title="Check Migration Status">
    Verify which migrations have been applied:
    
    ```bash
    masonite-orm migrate:status -C database/config.py -d database/migrations
    ```
  </Step>
  
  <Step title="Run Migrations">
    Execute pending migrations:
    
    ```bash
    masonite-orm migrate -C database/config.py -d database/migrations
    ```
  </Step>
</Steps>

<Warning>
  **Important:** Some migrations (like `refacto_members`) perform data migrations that can take time on large databases. Plan accordingly for production deployments.
</Warning>

## Rolling Back Migrations

To rollback the last batch of migrations:

```bash
masonite-orm migrate:rollback -C database/config.py -d database/migrations
```

<Note>
  Be cautious when rolling back migrations in production, as this may result in data loss.
</Note>

## Creating New Migrations

If you're extending FixTweetBot and need to create a new migration:

```bash
masonite-orm migration create_your_table -C database/config.py -d database/migrations
```

This will create a new migration file with the current timestamp.

## Troubleshooting

<AccordionGroup>
  <Accordion title="Connection Error">
    Verify your database credentials in `override.config.yml` and ensure the database server is running.
  </Accordion>
  
  <Accordion title="Migration Already Applied">
    Check the `migrations` table in your database to see which migrations have been run.
  </Accordion>
  
  <Accordion title="Foreign Key Constraint Failed">
    Ensure migrations are run in order. The timestamp prefix ensures correct ordering.
  </Accordion>
</AccordionGroup>
